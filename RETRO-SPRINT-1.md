\## Start: สิ่งที่ควรเริ่มทำในรอบต่อไป



\- เริ่มตรวจสอบ Acceptance Criteria และ Edge Case ร่วมกันก่อนลงมือเขียนโค้ด  

&#x20; เหตุผล: ใน Sprint นี้พบปัญหาชื่อไฟล์ `item.json` และ `items.json`

&#x20; ไม่ตรงกัน รวมถึงกรณีไฟล์ว่างและข้อมูลติดลบที่ต้องกลับมาแก้ภายหลัง



\- เริ่มเขียนและรันทดสอบกรณีปกติและกรณีผิดปกติก่อนเปิด Pull Request  

&#x20; เหตุผล: จะช่วยตรวจพบปัญหา เช่น การตัดสินค้าเกินสต๊อก

&#x20; การไม่พบสินค้า และ JSON มีรูปแบบไม่ถูกต้องก่อนส่งให้เพื่อน Review



\- เริ่มกำหนดชื่อไฟล์ ฟังก์ชัน และรูปแบบ Commit Message ให้เป็นมาตรฐานเดียวกัน  

&#x20; เหตุผล: ช่วยให้สมาชิกเข้าใจโค้ดได้ง่าย และลดความสับสนระหว่างไฟล์

&#x20; เช่น `list.py`, `inventory.py`, `item.json` 



\## Stop: สิ่งที่ควรหยุดทำ



\- หยุดรอให้เขียนโค้ดเสร็จหลายส่วนแล้วจึง Commit พร้อมกัน  

&#x20; ตัวอย่างใน Sprint นี้: บางช่วงมีการเปลี่ยนแปลงหลายอย่างอยู่ในไฟล์เดียวกัน

&#x20; ทำให้ตรวจสอบได้ยากว่าการแก้ไขแต่ละส่วนมีวัตถุประสงค์อะไร



\- หยุดทดสอบเฉพาะกรณีที่ข้อมูลถูกต้อง  

&#x20; ตัวอย่างใน Sprint นี้: ต้องกลับมาเพิ่มการตรวจสอบกรณีสินค้าไม่เพียงพอ

&#x20; ไฟล์ JSON หาย ไฟล์ว่าง และ JSON ไม่ถูกต้องภายหลัง



\- หยุดใช้ข้อเสนอแนะจาก AI แล้ว Commit ทันทีโดยไม่ตรวจและรันในเครื่อง  

&#x20; ตัวอย่างใน Sprint นี้: AI พบว่าชื่อไฟล์ในโค้ดเป็น `items.json`

&#x20; แต่ไฟล์จริงชื่อ `item.json` จึงต้องแก้และทดสอบอีกครั้งก่อน Merge



\## Continue: สิ่งที่ทำได้ดีและควรทำต่อ



\- ใช้ Feature Branch แยกตามงาน และไม่ Commit หรือ Push ตรงเข้า `main`



\- เปิด Pull Request ให้สมาชิกคนอื่นตรวจสอบก่อน Merge

&#x20; และกำหนดให้ Reviewer เป็นผู้ Merge แทนเจ้าของ PR



\- Commit งานเป็นก้อนเล็ก พร้อมข้อความที่ระบุประเภทและวัตถุประสงค์

&#x20; เช่น `feat:`, `fix:`, `test:` และ `chore:`



\- รันโปรแกรมจริงจาก branch `main` หลัง Merge

&#x20; เพื่อตรวจสอบว่าระบบทั้งหมดทำงานร่วมกันได้



\- Review โค้ดโดยพิจารณา Correctness, Readability, Edge Case

&#x20; และคุณภาพของ Commit Message ไม่ใช่เพียง Comment ว่า “LGTM”



\## AI Commit Audit



PR #25 เป็น PR ที่ต้องแก้ข้อความจาก AI หรือข้อความร่างเดิมมากที่สุด



ข้อความเดิมระบุเพียงชื่อ Task หรือบอกกว้าง ๆ ว่าเพิ่มฟังก์ชัน

แต่ยังไม่อธิบายว่าสร้างฟังก์ชันเพื่ออะไร และยังไม่ระบุพฤติกรรมสำคัญ

เมื่อไม่พบไฟล์หรือ JSON ไม่ถูกต้อง



ข้อความที่ปรับปรุงแล้วควรมีลักษณะ เช่น:



`feat: เพิ่มฟังก์ชัน load\_items() เพื่ออ่านข้อมูลสินค้าและรองรับไฟล์ JSON ที่ไม่ถูกต้อง`



นอกจากนี้ ข้อเสนอแนะจาก AI ยังต้องตรวจสอบกับไฟล์จริงใน repository

เพราะพบว่าชื่อไฟล์ที่โค้ดเรียกใช้ไม่ตรงกับชื่อไฟล์จริง
